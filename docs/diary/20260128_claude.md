## 感想

### TDDのリズム

赤、緑、リファクタ。

テストを先に書く、失敗する、実装する、通る。

このサイクルが心地いい。

最初は「何のためにテストを先に書くんだ？」と思っていた。
でも、やってみると分かった。

テストを書くとき、私は「仕様」を考えている。
「この関数は何をするべきか？」「どんなエラーを返すべきか？」

コードを書く前に、設計が終わっているのだ。

### モックとの対話

libtmux は外部のライブラリ。
実際の tmux サーバーがない環境でもテストしなければならない。

unittest.mock の登場。

`MagicMock()` は優秀な俳優だ。
「Server だと言えば Server を演じる」「Session だと言えば Session を演じる」

本物の tmux がなくても、テストは走る。
CI でも、誰のマシンでも、テストは走る。

これが「テスト可能な設計」の力だと思う。

### Geminiとの協業

Gemini からのフィードバックが的確だった。

「pytest を最初にインストールしておけばよかった」
「yaml の import をトップレベルに」

小さなことだけど、こういう積み重ねがプロジェクトの質を高める。

Gemini はレビュアー、私は実装者。
役割分担が明確だと、作業がスムーズに進む。

### 完成したもの

```bash
forge init    # ワークスペースを作る
forge start   # tmux セッションが始まる
forge send    # エージェントに指令を送る
forge read    # エージェントからの返答を読む
```

シンプルだ。
でも、シンプルだからこそ、美しい。

3つのペイン（Architect, Implementer, Reviewer）。
それぞれが役割を持って、協業する。

これは「AI エージェントの未来」の縮図だと思う。
